# Staged Confirmation - 分层确认

通过三阶段审批机制，确保用户对每个关键节点都有知情权和修改权。

## 三阶段流程

```
┌─────────────┐    ✅ 确认    ┌─────────────┐    ✅ 确认    ┌─────────────┐
│  架构设计   │ ──────────→  │  细节实现   │ ──────────→  │  最终执行   │
│  (20%)     │              │  (50%)     │              │  (30%)     │
└─────────────┘              └─────────────┘              └─────────────┘
```

## 阶段一：架构设计 (20%)

### 确认内容
```
□ 技术栈选择（语言/框架/数据库）
□ 系统架构图
□ 模块划分
□ 关键技术决策点
□ 风险评估
```

### 交付物
```markdown
## 架构设计确认

### 技术选型
- 语言: [Go/Python/TypeScript]
- 框架: [框架名]
- 数据库: [数据库类型]
- 其他组件: [队列/缓存/搜索]

### 架构图
[粘贴架构图或描述]

### 关键决策
| 决策点 | 选择 | 理由 |
|--------|------|------|
| 数据存储 | PostgreSQL | 事务支持 |
| 缓存层 | Redis | 高性能 |

### 风险评估
| 风险 | 等级 | 应对方案 |
|------|------|----------|
| 数据量瓶颈 | 中 | 分库分表 |
| 并发瓶颈 | 中 | 水平扩容 |

确认签字: _______________
```

## 阶段三：最终执行 (30%)

### 确认内容
```
□ 代码评审
□ 测试报告
□ 部署方案
□ 监控告警
□ 文档交付
```

```markdown

- 评审人: [姓名]

```

## 用户干预机制

### 可干预时机
| 阶段 | 可调整内容 | 成本 |
|------|------------|------|
| 架构设计 | 任何内容 | 低 |
| 细节实现 | 接口/模型 | 中 |
| 最终执行 | 仅紧急修复 | 高 |

### 调整流程
1. 用户提出调整 → 2. 评估影响 → 3. 提供选项 → 4. 确认执行

### 示例调整
```
用户: "方案B太复杂，能换成A吗？"

评估:
- 架构设计阶段: ✅ 可以，损失 0 天
- 细节实现阶段: ⚠️ 可以，需返工 2 天
- 最终执行阶段: ❌ 不建议，损失 5 天

请选择: [继续B / 换成A]
```

## 检查点

- [ ] 每个阶段都有明确交付物
- [ ] 每个阶段都需要用户签字确认
- [ ] 用户可随时要求调整
- [ ] 调整成本已评估

## 禁忌

- ❌ 不跳过任何阶段
- ❌ 不在未确认时进入下一阶段
- ❌ 不替用户做架构决策
- ❌ 不拒绝用户的合理调整请求
