# 接收代码审查 (Receiving Code Review)

## 概述

如何专业地接收和处理代码审查反馈，提高协作效率和个人成长。

## 心态建设

### 正确的审查态度
```
┌─────────────────────────────────────────────────────────┐
│                    审查反馈心态                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ✅ 代码被审查 ≠ 个人能力被否定                           │
│  ✅ 审查是为了提高代码质量和团队协作                       │
│  ✅ 审查者是帮助者，不是批评者                            │
│  ✅ 每次审查都是学习和改进的机会                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 处理流程

### 1. 收到审查请求
```markdown
## 审查通知处理

- [ ] 查看审查内容
- [ ] 理解审查意见
- [ ] 分类处理意见
- [ ] 安排时间处理
```

### 2. 阅读审查意见
```markdown
## 理解审查要点

### 问题分类
🔴 严重问题 - 必须修复
🟡 建议改进 - 建议采纳
🟢 讨论事项 - 可以讨论

### 优先级排序
1. 阻塞性问题（影响功能）
2. 安全性问题
3. 性能问题
4. 代码规范问题
5. 优化建议
```

### 3. 回复审查意见
```markdown
## 回复模板

### 同意修改
👍 好的，我这就改

### 部分同意
👀 这个我理解，但考虑到...，建议...

### 需要讨论
🤔 这个地方我有些疑问，能否线下沟通？

### 不同意
💭 我的考虑是...，觉得当前实现也合理，原因...
```

## 常见问题处理

### 问题 1: 不理解审查意见
```markdown
# 正确做法
"不好意思，这个地方我没理解清楚，能否详细解释一下？
或者能不能给一个具体的示例说明？"

# ❌ 错误做法
"我觉得这样也可以啊"  (没有真正理解)
```

### 问题 2: 不同意审查意见
```markdown
# 正确做法
"我理解你的顾虑，但我的考虑是：
1. 性能方面...
2. 可维护性方面...
3. 当前场景的特殊性...

综合来看，我建议保持现状，你觉得呢？"

# ❌ 错误做法
- 直接忽略继续推进
- 情绪化反驳
- 强行要求通过
```

### 问题 3: 审查意见太多太碎
```markdown
# 正确做法
"感谢仔细审查！我注意到有很多小问题，
我计划：
1. 先修复所有严重问题
2. 再统一处理这些建议
预计今天下班前可以完成"

# 可以请求分类
"这些意见能否帮忙分个类？
哪些是必须修改的，哪些是建议修改的？"
```

## 修复技巧

### 1. 小步提交
```bash
# 每个问题单独提交，便于审查
git add file1.js
git commit -m "fix: 修复空指针问题"

git add file2.js
git commit -m "fix: 添加参数校验"
```

### 2. 回复格式
```markdown
## 修复格式

### 已修复
[文件:行号] - 已修改为...

### 需要讨论
[文件:行号] - 我的理解是...，原因...

### 拒绝原因
[文件:行号] - 经过评估，我认为不需要修改，因为...
```

### 3. 标记完成
```markdown
## 审查意见状态

| 序号 | 问题 | 状态 | 说明 |
|------|------|------|------|
| 1 | 空指针风险 | ✅ 已修复 | 已添加 null 检查 |
| 2 | 变量命名 | ✅ 已修复 | 已改为有意义命名 |
| 3 | 代码重复 | 🔄 处理中 | 计划重构 |
| 4 | 注释缺失 | 💭 待讨论 | 想听听你的看法 |
```

## 与审查者沟通

### 时机选择
- 简单问题 → 直接在审查界面回复
- 复杂问题 → 约时间线上讨论
- 争议问题 → 记录讨论结果并发邮件确认

### 沟通话术
```markdown
# 请求重新审查
"我已经根据审查意见完成了修改，麻烦重新审查一下"

# 请求讨论
"这个问题我想详细聊聊，今晚有时间吗？"

# 更新进展
"审查意见比较多，我计划分两批完成：
第一批（严重问题）：今天下午
第二批（优化建议）：明天上午"
```

## 成长建议

### 1. 记录学习
```markdown
## 审查学习记录

### 从这次审查学到的
1. 要注意边界条件处理
2. 重要方法要加注释
3. 变量命名要准确反映用途

### 下次改进
- 提交前先自己 review 一遍
- 写完代码先自测
```

### 2. 主动学习
- 理解审查意见背后的原理
- 学习代码规范和最佳实践
- 理解不同场景的权衡取舍

### 3. 感恩反馈
```markdown
"感谢仔细的审查，学到了很多！"
```
