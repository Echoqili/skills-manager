# 请求代码审查 (Requesting Code Review)

## 概述

如何有效地发起代码审查请求，提高审查效率和问题发现率。

## 发起审查前

### 自我检查清单
```
┌─────────────────────────────────────────────────────────┐
│                  提交审查前自检                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ☐ 代码可以正常运行                                       │
│  ☐ 所有测试通过                                          │
│  ☐ 没有 console.log/debugger 残留                        │
│  ☐ 代码格式正确                                          │
│  ☐ 变量命名有意义                                        │
│  ☐ 关键逻辑有注释                                        │
│  ☐ 关联需求已实现                                        │
│  ☐ 更新了相关文档（如有必要）                             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 本地预审
```bash
# 运行所有测试
npm test

# 运行 lint
npm run lint

# 构建检查
npm run build

# 自我 Code Review
git diff origin/dev...HEAD
```

## 创建 Pull Request

### PR 标题规范
```
[类型] 简短描述

示例:
[feat] 添加用户手机号登录功能
[fix] 修复支付回调重复处理问题
[refactor] 重构订单查询服务
```

### PR 描述模板
```markdown
## 变更内容

### 新增功能
- 功能点 A
- 功能点 B

### 修改内容
- 修改点 A
- 修改点 B

### 修复问题
- 问题 A
- 问题 B

## 变更原因

为什么要做这个变更？

## 影响范围

这个变更会影响哪些功能/模块？

## 测试情况

- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 手动测试通过

## 相关链接

- Issue: #123
- 设计文档: [链接](url)

## 截图（如有 UI 变更）

[截图]
```

## 选择审查者

### 优先选择
```markdown
1. 功能的直接负责人
2. 相关模块的维护者
3. 有相关经验的同事
4. 代码规范熟悉者
```

### 避免
- 同时请求太多人（2-3 人为宜）
- 请求不相关领域的专家
- 请求正在忙碌的同事

## 审查过程中

### 及时响应
```markdown
## 响应承诺

"我会尽快处理审查意见，预计在 X 小时内完成"

## 更新状态

如有长时间无法响应的情况：
"抱歉临时有事，处理时间调整为明天上午"
```

### 保持透明
```markdown
## 进展更新

### 已完成
- [x] 问题 1 的修复
- [x] 问题 2 的修复

### 进行中
- [ ] 问题 3 正在处理

### 遇到困难
"问题 3 比较复杂，可能需要讨论"
```

## 催促审查

### 合适的催促
```markdown
"您好，方便的时候帮忙审查一下这个 PR 吗？
这个功能比较紧急，团队在等这个 PR 合并。

如果时间不方便，请告诉我什么时候合适，谢谢！"
```

### 避免
- 频繁催促
- 在不同时区同事的深夜催促
- 语气强硬或带有情绪

## 合并时机

### 可以合并的情况
```markdown
✅ 所有审查意见已处理
✅ 所有测试通过
✅ 至少一人 Approve
✅ 没有阻塞性问题
```

### 需要等待的情况
```markdown
⏳ 还有未处理的审查意见
⏳ 测试失败需要修复
⏳ 需要更多测试覆盖
⏳ 等待其他关联 PR 合并
```

## 特殊情况

### Draft PR
```markdown
# 适合场景
- 还在开发中，需要提前反馈
- 有很多问题需要讨论
- 不确定方向是否正确

## 转换流程
Draft PR → 准备好后 → Ready for Review
```

### WIP (Work In Progress)
```markdown
# PR 标题格式
WIP: [feat] 用户登录功能

# 或使用 Draft PR 功能
```

## 最佳实践

1. **小而频繁**: 多次小 PR 比一次大 PR 更容易审查
2. **描述清晰**: 让审查者快速理解意图
3. **自审优先**: 自己先 review 一遍
4. **及时响应**: 尽快回复审查意见
5. **感谢反馈**: 对审查者的付出表示感谢
