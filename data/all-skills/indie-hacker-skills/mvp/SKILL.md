# MVP

## Purpose

构建最小可行产品，砍掉所有非生存必需的功能。核心是**用最少的代码验证最大的假设**，快速迭代而非一步到位。

**适用场景：**
- 第一次做产品，不知道用户真正需要什么
- 资源有限（时间/金钱/技术）
- 需要快速验证市场反应
- 担心投入太多做出来没人用

## Key Concepts

### MVP vs MMB (Minimum Marketable Feature)

```
MVP = 最少功能验证假设
MMB = 最少功能满足市场

案例：邮件订阅工具
- MVP：只能订阅，不能批量管理
- MMB：订阅+标签+自动化+分析

时序：先MVP验证 → 再MMB赚钱
```

### 功能优先级矩阵

| 功能 | 用户痛(1-5) | 开发成本(1-5) | 优先级 |
|------|------------|--------------|--------|
| 核心价值 | 5 | 1 | 🔴 必须做 |
| 核心价值增强 | 4 | 2 | 🔴 必须做 |
| 减少摩擦 | 3 | 2 | 🟡 考虑 |
| 扩展功能 | 2 | 3 | 🟢 可选 |
| 炫酷但无用 | 1 | 5 | ❌ 不做 |

### MVP 死亡线

```
砍功能检查三问：
1. 删掉这个功能，用户还会付钱吗？
   → 会：删
   → 不会：保留

2. 没有这个功能，产品能解决问题吗？
   → 能：删
   → 不能：保留

3. 这个功能多久后会有用？
   → 6个月后：删
   → 下周就需要：保留
```

### 2. Gumroad 案例拆解

```
Gumroad MVP (2011年)：
□ 支付按钮
□ 文件上传
□ 确认邮件
□ 下载链接
□ 收款到账户

砍掉的功能：
× 订阅功能 → 后来加
× 优惠券系统 → 后来加
× 分析面板 → 后来加
× API接入 → 后来加
× 批量上传 → 后来加

结果：2周开发 → 立即收款验证
```

## Examples

### 成功MVP案例

| 产品 | MVP版本 | 砍掉的功能 | 结果 |
|------|---------|-----------|------|
| Dropbox | 视频+概念 | 真实App | 7.5万等待列表 |
| Zapier | 人工串联 | 自动化 | 验证需求后开发 |
| Airbnb | 静态页面+租气垫 | 支付、地图 | 验证后再做 |
| Groupon | 手动物流 | 自动发货 | 验证后开发 |

### 失败MVP案例

```
产品：项目管理工具
功能：甘特图+资源管理+自动化+API+...

问题：开发18个月
结果：市场已变化，Notion已成标准

学到的：做太多=做太久=错过时机
```

## Common Pitfalls

1. **功能蔓延**：总觉得"这个也很重要"
2. **完美主义**：觉得需要"像个正经产品"
3. **忽视核心**：做了很多锦上添花，忘了核心价值
4. **过度优化**：在错误的方向上快速迭代
5. **过早规模化**：没有验证就考虑服务器架构

## Decision Checklist

- [ ] 功能列表≤5个
- [ ] 每个功能都通过"删掉测试"
- [ ] 开发时间≤2周
- [ ] 有真实用户在等待
- [ ] 可以立即收款

## Metrics

```
MVP成功标准：
□ 1周内获得10个付费用户
□ 用户愿意主动推荐
□ 付费原因与预期一致
□ 至少1个"这个刚好解决我的问题"

MVP失败信号：
□ 用户问"你们有没有XX功能"
□ 需要大量客服解释
□ 抱怨缺少核心功能
□ 没人愿意付费
```
